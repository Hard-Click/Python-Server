# 문제 정의: Ground Truth 없는 개인화 파라미터를 실사용자 없이 어떻게 검증할 것인가

## 1. 배경

Python-Server(돼지엄마 AI 스케줄러)는 학생의 실제 학습 소요시간을 관측해서, 강사가 잡은
고정 추정치(`expected_duration_min`)를 그 학생 기준으로 보정하는 **효율계수**를 CP-SAT
스케줄링에 반영한다(`domain/scheduler.py::compute_efficiency_coefficient`).

이때 관측 페이스(raw)를 100% 그대로 믿지 않고, 강사 추정치(1.0) 쪽으로 일정 비율만
당기는 **축소추정(shrinkage)** 방식을 쓴다:

```
효율계수 = 1.0 + stretch_factor × (raw − 1.0)
```

`stretch_factor`는 현재 0.5(고정 상수)다.

## 2. 제약 조건

문제를 풀기 전에 아래 조건을 고정으로 두고 시작한다 — 이 조건 중 하나라도 무시한
해법(예: 실사용자 대상 A/B, 복잡한 온라인 학습 모델)은 지금 시점에는 채택 불가하다.

1. **실사용자 트래픽 전무**: 서비스 오픈 전이라 학생 계정도, 로그도, 노출 이력도 0건.
   설문·인터뷰 대상도 없다.
2. **리소스 제한**: 팀/해커톤 규모 프로젝트라, 전담 데이터 분석 인력이나 몇 주 단위
   실험 운영 여력이 없다. 발표 마감이라는 하드 데드라인이 있다.
3. **채택 방식은 이미 확정**: `stretch_factor`를 쓴 선형 축소추정(shrinkage) 공식
   자체는 바꾸지 않는다 — 이번 문제는 "이 공식이 맞냐"가 아니라 "이 공식 안의 상수값을
   지금 상황에서 어떻게 정당화하냐"다.
4. **명시적으로 스코프 아웃한 것들**: Multi-armed Bandit, 신뢰도 이론(credibility
   theory) 기반 동적 가중치, Offline Policy Evaluation(IPS/SNIPS/DR), 대규모 사용자
   설문, LLM 에이전트 시뮬레이션 — 전부 이유는 다르지만(적용 불가/순환논증/오버스펙)
   지금 채택하지 않는다.

## 3. 구체적 문제

`stretch_factor`를 왜 0.5로 정했는지, 그리고 이 값이 실제로 적절한지를 무엇으로
검증할 것인가?

## 4. 이 문제가 일반적인 ML 파라미터 튜닝과 다른 이유 — Ground Truth 부재

일반적인 지도학습 문제는 정답 레이블이 있어서 파라미터를 그 레이블에 맞춰 최적화하면 된다.
그런데 이 문제는 구조적으로 정답이 없다:

- 관측되는 건 "학생이 강의 하나에 실제로 얼마나 걸렸나(raw pace)"뿐이다.
- 하지만 그 뒤에 숨은 **원인**은 관측 불가능하다 — 같은 raw=1.5배 느림이라도, ①진짜
  이해가 느려서 느린 학생과 ②집중을 안 해서 오래 걸린 학생(사실은 밀어붙이면 더 빠를 수
  있는 학생)을 구분할 방법이 없다.
- 이 두 유형은 `stretch_factor`를 세게 줬을 때(관측치를 안 믿고 1.0 쪽으로 강하게
  당길 때) **정반대로 반응한다**: ①번 학생은 무리한 페이스 강요로 완주가 무너지고,
  ②번 학생은 오히려 실제 능력에 맞는 페이스를 되찾아 좋아진다.
- 즉 "정답 stretch_factor"는 존재하지 않고, 존재한다 해도 **모집단에서 두 유형의
  비율**에 따라 달라지는데 그 비율 자체를 알 방법이 없다(직접 관측 불가능한 잠재변수).

## 5. 왜 표준 검증 방법들이 지금 이 상황에 안 맞는가

| 방법 | 왜 지금 못 쓰는가 |
|---|---|
| A/B 테스트(실사용자) | 서비스 오픈 전이라 실사용자 트래픽이 0명 - 애초에 노출시킬 대상이 없음 |
| 오프라인 정책평가(IPS/SNIPS/DR) | 과거 정책의 로그와 노출확률(propensity)이 전제인데, 로그 자체가 없음(원천적으로 적용 불가, "복잡해서 안 쓴다"가 아니라 "적용 조건 자체가 성립 안 함") |
| 사용자 설문 | 설문할 실사용자가 없음 |
| 합성 데이터로 목적함수 최적화 | 정답(ground truth)을 우리가 만들고, 그 정답 기준으로 우리가 채점해서, 우리가 만든 가정에 가장 잘 맞는 값을 "발견"하는 순환논증 — 결론이 가정을 그대로 되비출 뿐 실증적 근거가 아님 |
| LLM 에이전트 시뮬레이션 | 형태만 다를 뿐 위 순환논증과 동일한 구조(사람이 정한 "성향 변수"가 곧 숨겨둔 정답) |
| Bayesian 신뢰도 이론 / Multi-armed Bandit | 통계적으로는 더 원칙 있는 접근이지만, 지금 리소스(오픈 전 해커톤/팀 프로젝트 규모)로는 구현·운영 비용이 문제 규모에 안 맞음(오버엔지니어링) |

## 6. 그래서 실제로 풀어야 하는 질문 (재정의)

"stretch_factor의 최적값은 얼마인가"가 아니라:

1. **지금 고정 상수 0.5를 쓰는 것이 이 상황(데이터 0, 리소스 제한)에서 합리적인 선택인가?**
   → 근거: 복잡한 동적 모델도 검증 데이터가 없긴 마찬가지라 자유도만 늘어남(거짓 정밀도),
   이 프로젝트 전체의 "콜드스타트엔 단순하게" 원칙과 일관됨, 나중에 교체 비용이 낮게
   설계돼 있음(파라미터화·실험 배정 인프라 이미 존재).
2. **이 공식(선형 축소추정) 자체가 코드적으로 안전하게 동작하는가?**
   → 답: 단조성·경계값·무관성 등 속성 테스트로 검증 완료(`tests/test_domain.py`) - 이건
   ground truth 없이도 검증 가능한 부분.
3. **sf가 실제 운영 결정을 얼마나, 어떤 구조로 바꾸는가?** (특정 값을 고르는 질문이 아님)
   → 검증 방식을 바꿨다. 초기엔 합성 outcome 점수로 variant 승자를 뽑는 스크립트
   (`scripts/simulate_stretch_factor_ab_test.py`)를 썼으나, 이건 "coeff가 우리가 심어둔
   true_ratio에 가까울수록 좋다"는 손실을 정의로 박아둔 순환논증이라(5절 참조) 폐기한다.
   대신 관측 불가능한 true_ratio를 일절 안 쓰고 **관측 가능한 입력만** 실제 프로덕션
   함수(`compute_efficiency_coefficient`/`compute_required_extension_weeks`/
   `compute_slip_status`)에 통과시켜 **sf가 실제 운영 결정을 얼마나 바꾸는지**만 재는
   결정민감도 분석으로 대체한다(`scripts/analyze_stretch_factor_decision_sensitivity.py`).
   이 분석은 **baseline(기준값)도 두지 않는다** — 특정 sf(예: 0.5)를 원점에 놓으면 측정이 그
   값을 중심으로 편향되기 때문(실제로 초기 "0.5 대비" 버전은 무변화 구간을 `[0.45,0.55]`로
   냈는데, baseline을 없애자 평탄 구간이 고sf 쪽으로 옮겨갔다 — 그 자체가 편향의 증거였다).
   측정 결과(1400명 합성 모집단, reference-free, 부호·순위만 신뢰):
   - **56.4%의 학생은 sf를 0.0~1.0 어디에 놔도 이산결정(ext/push)이 동일**하다 — 절반 이상은
     sf 값 선택 자체가 운영 레버리지 밖.
   - **콜드스타트(완료<3건)/마감임박(D-100 이내) 버킷은 sf와 무관하게 결정이 고정**된다
     (전자는 coeff=1.0 강제, 후자는 push_mode 강제).
   - sf가 실제로 결정을 가르는 건 소수 스트레스 버킷(심각한 밀림 → 연장, raw~1.4 → push)뿐이다.
   - 이산결정 국소 민감도는 저sf에서 크고 고sf에서 평평하다 — 특정 값이 특별히 안전하다는 신호는
     없다. `weekly_minutes`(보조·근사치)는 coeff에 선형으로 붙어 움직여 sf는 "부하 다이얼"에 가깝다.
   → 결론: **어떤 sf 값도 이 분석이 채택/추천/기각하지 않는다.** "0.5가 최적/강건한 선택"이라는
   주장도 하지 않는다(0.5는 실험 근거 없이 정해진 값이라 특별 지위 없음). 여기서 말할 수 있는 건
   딱 "sf의 운영 레버리지가 좁고 그 구조가 어떻게 생겼나"까지다. 값의 채택/기각은 반례(Phase 1)와
   worst-case(Phase 2) 이후 [roadmap_stretch_factor_offline.md](roadmap_stretch_factor_offline.md)에서 다룬다.
   (caveat: `weekly_minutes`·누적 slip은 CP-SAT를 돌리지 않은 파생 근사치라 절대값 인용 금지.)
4. **실사용자가 생기면 언제·무엇을 기준으로 이 상수를 재검증할 것인가?**
   → 졸업 기준을 수치로 명시(`docs/policy_constants.md`): variant당 30명 이상, 4주 이상
   활동, 완주율/성적 추이 로깅 존재 — 이 세 조건을 채우기 전 나온 비교 결과는 노이즈로
   간주하고 의사결정에 안 씀.

## 7. 결론

Ground truth가 없는 문제에서 "값의 최적성"을 증명하려는 시도는 필연적으로 순환논증에
빠진다. 대신 검증 가능한 건 ①설계의 구조적 타당성(왜 이 형태를 골랐는가), ②코드 레벨
안전성(속성 테스트), ③불확실성 하에서의 강건성(민감도 분석), ④실사용자 확보 후 재검증
계획(졸업 기준)이다 — 이 넷을 갖추는 것이 "0.5가 정답이다"를 주장하는 것보다 훨씬
방어 가능하고 정직한 결과물이다.
